# Phase 10 설계: 콘솔 한글 출력 깨짐 수정

`PLAN.md`의 Phase 10 목표(Windows 콘솔에서 한글이 깨져 보이는 문제 해결)에 대한 상세 설계. 사용자가
실제 실행 환경에서 한글이 제대로 표시되지 않는 문제를 보고하여 신설한 버그 수정 Phase다.

## 원인 분석

- 소스 파일(`.cpp`)의 한글 문자열 리터럴은 UTF-8로 저장되어 있고, `std::printf`는 이 바이트를 그대로
  표준출력에 내보낸다.
- Windows 콘솔(`conhost.exe`)은 프로세스 시작 시 **활성 코드페이지**를 기준으로 출력 바이트를 해석해
  화면에 렌더링한다. 한국어 Windows의 기본 활성 코드페이지는 보통 CP949(또는 로캘에 따라 CP437 등)이며,
  "전 세계 언어 지원을 위한 유니코드(UTF-8) 사용" 옵션이 켜져 있지 않으면 UTF-8(65001)이 아니다.
- 코드페이지가 UTF-8이 아닌 상태에서 UTF-8 바이트가 그대로 출력되면, 콘솔이 이를 다른 인코딩으로 잘못
  해석해 한글이 깨져 보인다(mojibake). 지금까지 개발 중 확인에 사용한 Git Bash(mintty) 터미널은 자체적으로
  UTF-8을 항상 사용하므로 이 문제가 드러나지 않았다 — 실제 사용자의 cmd.exe/PowerShell 콘솔에서만
  나타나는 환경 종속 버그였다.

## 확정된 결정 사항

### `SetConsoleOutputCP`/`SetConsoleCP`로 코드페이지를 UTF-8로 고정

`main()` 시작 시 Windows API로 콘솔의 출력·입력 코드페이지를 모두 UTF-8(65001)로 강제 설정한다.

```cpp
// src/main.cpp — 최상단에 추가
#ifdef _WIN32
#include <windows.h>
#endif

int main() {
#ifdef _WIN32
    SetConsoleOutputCP(CP_UTF8);  // 출력: printf로 내보내는 UTF-8 바이트를 올바르게 렌더링
    SetConsoleCP(CP_UTF8);        // 입력: 고객명/검색어 등 한글 입력도 올바르게 해석
#endif
    ...
}
```

- `#ifdef _WIN32`로 감싸 이식성을 해치지 않는다 (이 프로젝트는 Windows/MinGW 전용 빌드 환경이지만,
  Windows가 아닌 환경에서 우연히 빌드될 경우에도 컴파일이 깨지지 않도록 방어).
- 이미 코드페이지가 UTF-8인 환경(Git Bash, 또는 "Beta: UTF-8 전 세계 언어 지원" 옵션이 켜진 Windows
  11)에서는 아무 효과가 없는 멱등적 호출이다 — 회귀 위험이 없다.
- 다른 파일(`ConsoleView`, `MainController` 등)은 변경하지 않는다. 문제의 근본 원인이 "콘솔 자체의
  코드페이지 설정"이므로, 애플리케이션 진입점 한 곳에서 한 번만 고치면 모든 출력·입력에 일괄 적용된다.

## 검증 방법의 한계와 대안

이 버그는 **콘솔이 바이트를 화면에 렌더링하는 방식**의 문제이므로, 표준출력을 파이프로 리다이렉트해
캡처하는 자동화 도구(Bash/PowerShell 도구를 통한 실행)로는 재현·확인이 불가능하다 — 리다이렉트된 스트림은
콘솔의 코드페이지와 무관하게 원본 UTF-8 바이트 그대로 캡처되어, 이미 UTF-8을 아는 도구로 다시 보여주면
항상 정상으로 보인다. 실제 사용자가 겪는 현상은 실행 파일을 콘솔 창에 직접 붙여(attach) 실행할 때만
나타난다.

따라서 이번 Phase의 검증은 다음 두 가지로 대신한다.

1. **간접 증거**: `chcp 949`로 콘솔 코드페이지를 강제 변경한 직후, 앱을 실행하고 종료한 뒤 같은 콘솔에서
   다시 `chcp`를 확인한다. 수정 전에는 그대로 949로 남아있고, 수정 후에는 앱이 `SetConsoleOutputCP`를
   호출했으므로 65001로 바뀌어 있어야 한다 — API 호출이 실제로 효과를 냄을 증명한다.
2. **회귀 테스트**: 기존 `ctest` 전체가 그대로 통과하는지 확인한다 (Windows API 호출은 콘솔 앱
   실행(`main.cpp`)에만 영향을 주고, 테스트 바이너리(`sample_order_tests.exe`)는 `main()`을 거치지
   않으므로 영향받지 않는다).

## 디렉토리/파일 변경 요약

| 파일 | 변경 |
|---|---|
| `src/main.cpp` | 최상단에 `#ifdef _WIN32` / `<windows.h>` 추가, `main()` 시작 시 `SetConsoleOutputCP`/`SetConsoleCP` 호출 추가 |

## Phase 10에서 하지 않는 것

- 콘솔 폰트가 한글 글리프를 지원하지 않는 경우(레거시 래스터 폰트)는 코드로 해결할 수 없는 별도 이슈이므로
  다루지 않는다 — 코드페이지를 UTF-8로 맞추는 것까지가 애플리케이션이 책임질 수 있는 범위다.
- 새 기능·도메인 로직 변경 없음.

## 완료 기준

- 기존 70개 테스트 전체 회귀 없이 통과 (`ctest`)
- `chcp 949` 강제 후 앱 실행·종료 시 콘솔 코드페이지가 65001로 바뀌는지 확인 (간접 증거)
- `main.cpp`에 코드페이지 고정 로직이 정확히 1곳에 존재하는지 육안 확인
